# Lab: Boot-Time Memory Detection (启动阶段内存检测)

[English](../../README.md)

难度: ★★★★☆

## 设计初衷

xv6 的物理内存范围是硬编码的：`PHYSTOP = 0xE000000`（224MB），`kinit1/kinit2` 直接把 `[end, PHYSTOP)` 之间的所有物理页加入空闲链表。这在 QEMU 中运行良好，因为 QEMU 精确提供 128MB 或 256MB 内存。

但在真实 PC 上，物理内存布局远非连续。ACPI、显卡 ROM、ISA 空洞等占据低地址空间的多个区间。如果内核盲目地把 `[0, PHYSTOP)` 当作可用内存，就会覆盖硬件保留区域，导致崩溃。

真实操作系统（Linux、Windows）在启动时通过 **BIOS 中断 `int 0x15/E820`** 查询可用内存映射。这个中断只能在**实模式**下调用——一旦切到保护模式就再也无法调用 BIOS 了。

本实验在 bootasm.S 切换保护模式之前插入内存检测代码，将结果传递给内核，让 `kalloc.c` 根据实际内存映射初始化空闲链表。

## 前置知识

- **BIOS int 0x15/E820**: 实模式下查询系统内存映射的标准接口。每次调用返回一个地址范围描述符，通过 `CF` 标志和 `EBX` 续传值控制迭代
- **实模式 vs 保护模式**: BIOS 中断只在实模式（16 位）下可用。bootasm.S 一旦 `CR0.PE = 1` 就再也不能调 BIOS 了
- **PC 内存布局中的"空洞"**:
  ```
  0x00000000 ~ 0x0009FFFF   640KB 常规内存（可用）
  0x000A0000 ~ 0x000BFFFF   VGA 显存（保留）
  0x000C0000 ~ 0x000FFFFF   BIOS ROM / ISA 空洞（保留）
  0x00100000 ~ ...           扩展内存（可用，但可能有 ACPI 等保留区间）
  ```
- **bootloader → 内核数据传递**: bootloader 和内核之间没有函数调用接口，只能通过约定内存地址或寄存器传递数据

## 实验内容

### 1. 理解当前 kalloc 的硬编码

阅读 `kernel/kalloc.c` 的 `kinit1()` 和 `kinit2()`，理解它们如何使用 `PHYSTOP` 和 `freerange`。思考：如果实际物理内存不是 224MB 会发生什么？

### 2. 在 bootasm.S 中调用 E820 (修改 boot/bootasm.S)

在切换保护模式之前（`CR0.PE = 1` 之前），插入 E820 检测循环。

**E820 调用协议**：
- 输入：`EAX=0xE820`，`EBX=0`（首次调用），`ECX=buffer size`，`EDX=0x534D`('SM')，`ES:DI=缓冲区地址`
- 输出：`CF=0` 成功，`EBX` 续传值（0 表示结束），缓冲区被填入一个 24 字节的条目

**需要实现**：
- 循环调用 `int 0x15`，每次成功后推进 DI 指针，直到 EBX=0 或 CF=1
- 将条目数量保存在约定位置（如 `0x7FFE`），条目数据保存在 `0x8000` 开始的缓冲区

**E820 条目结构**（每个 24 字节）：
- `+0`  uint32 addr_low / addr_high — 基地址
- `+8`  uint32 len_low / len_high — 长度
- `+16` uint32 type — 1=可用, 2=保留, 3=ACPI, 4=NVS, 5=缺陷

**地址选择**：`0x8000` 在 bootloader 下方、BDA 上方，不被任何启动代码使用。

### 3. 定义内核侧数据结构 (修改 include/memlayout.h)

**需要定义**：
- `struct e820_entry` — 对应 E820 返回的条目格式
- `E820_MAP_ADDR`、`E820_COUNT_ADDR` — 约定的内存地址常量
- `E820_TYPE_USABLE` 等类型常量
- `get_e820_map()` / `get_e820_count()` — 通过 P2V 访问的辅助函数

### 4. 修改 kalloc.c 使用 E820 映射 (修改 kernel/kalloc.c)

**需要实现** `kinit_e820()`：
- 遍历 E820 条目，只处理 `type == E820_TYPE_USABLE` 的条目
- 用 P2V 转换物理地址到虚拟地址
- 页对齐后跳过内核代码区域 `[end, ...)`，对剩余范围调用 `freerange`

### 5. 修改 main.c 初始化序列 (修改 kernel/main.c)

用 `kinit_e820()` 替代原来的 `kinit2()` 调用。注意：必须在 `kvmalloc()` 之后调用，因为需要完整的虚拟地址映射。

## 涉及的 OS 概念

| 概念 | 在本实验中的体现 |
|------|----------------|
| BIOS 系统调用 | `int 0x15/E820` 是实模式下查询硬件信息的标准接口 |
| 物理内存映射 | PC 内存并非连续可用，有 VGA、ROM、ISA 空洞等保留区 |
| 实模式 vs 保护模式 | BIOS 中断只在实模式可用，必须在 bootasm.S 早期完成 |
| Bootloader-内核协议 | 通过约定内存地址传递数据，类似 Linux 的 `boot_params` |
| 内存分配器 | kalloc 从硬编码范围 → 基于硬件检测的动态范围 |
| 页对齐 | 物理内存分配器只操作页对齐的地址 |

## 验证

### 编译和运行

```bash
make clean && make qemu-nox
```

### 验证目标

| 目标 | 预期行为 | 观察方式 |
|------|---------|---------|
| E820 检测成功 | 返回多个条目 | 打印函数输出条目数和内容 |
| 可用内存正确 | 只有 type=1 被加入空闲链表 | kinit_e820 日志 |
| 跳过保留区域 | VGA/ROM/ISA 空洞不被分配 | 不同内存大小下不崩溃 |
| 多内存大小适配 | 自动适配 64MB/128MB/256MB | `QEMUEXTRA="-m 64"` 测试 |

### 多内存大小测试

```bash
make qemu-nox QEMUEXTRA="-m 256"
make qemu-nox QEMUEXTRA="-m 64"
```

### 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| E820 返回 0 条目 | 调用时机不对（已在保护模式） | 确保在 `CR0.PE=1` 之前执行 |
| 数据全是零 | 缓冲区地址被覆盖 | 检查 0x8000 是否与 entryother 冲突 |
| kinit panic | E820 范围包含内核代码 | 跳过 `[end, ...)` 区域 |
| bootblock 超大 | E820 汇编代码太多 | 紧凑编码，放在 A20 之前 |

## 关键代码路径

- E820 检测: boot/bootasm.S → 实模式下 `int 0x15` 循环调用
- 数据存储: boot/bootasm.S → 条目写入 0x8000，计数存 0x7FFE
- 数据读取: include/memlayout.h → P2V 转换后访问
- 内存初始化: kernel/kalloc.c → 遍历 E820 条目，仅 type=1 调用 freerange

## 设计权衡

| 方面 | 硬编码 PHYSTOP (原始) | E820 动态检测 (本实验) |
|------|---------------------|---------------------|
| 可移植性 | 仅适用于固定内存大小的 QEMU | 适用于任何 PC |
| 复杂度 | 极简 | 需要 BIOS 调用 + 数据传递 |
| 安全性 | 可能在真实 PC 上损坏硬件 | 正确跳过保留区域 |
| 灵活性 | 改内存大小需重编译 | 自动适配 |
