# Lab: Heap Allocator (用户空间堆内存分配器)

[English](../../README.md)

难度: ★★★☆☆

## 设计初衷

xv6 的用户程序目前通过 `malloc()`/`free()` 管理堆内存，但底层实现极其简单——`malloc` 仅调用 `sbrk` 扩展堆，从不回收；`free` 是空操作。这意味着程序无法重用已释放的内存，长时间运行必然耗尽地址空间。

本实验要求用 **显式空闲链表 (explicit free list)** 实现一个真正可用的分配器：

- **分配**: First-Fit 策略在空闲链表中找到第一个足够大的块
- **释放**: 将块归还链表，并与相邻空闲块合并（Coalescing）
- **扩展**: 空闲链表中找不到合适块时，调用 `sbrk` 向内核申请新页

核心问题：*"free(p) 之后，这块内存怎么知道自己有多大、该放回哪里？"*

## 前置知识

- **`sbrk` 系统调用**: 调整进程数据段上限（heap break），返回旧的 break 地址。`sbrk(n)` 向上扩展 n 字节，`sbrk(0)` 返回当前 break
- **隐式头部 (Block Header)**: 分配器在每块内存前面藏一个 header，记录块的大小和分配状态
- **内存对齐**: x86 要求数据按 4 字节对齐，分配器需要将所有请求上舍入到对齐粒度
- **碎片化**: 内部碎片（块比请求大）和外部碎片（空闲块太小而不连续）

```
堆内存布局:
┌──────┬─────────────────┬──────┬─────────────────┬──────┐
│ HDR  │   用户数据区     │ HDR  │   用户数据区     │ ...  │
│8字节 │   (payload)     │8字节 │   (payload)     │      │
└──────┴─────────────────┴──────┴─────────────────┴──────┘
  ↑ malloc 返回的指针指向 payload 起始，HDR 在其前面
```

## 实验内容

### 1. 设计块头部结构 (lib/xv6_stdlib.h)

定义描述每个内存块的元数据结构：

```c
typedef struct block_hdr {
    uint   size;          // 本块总大小（含 header），以字节计
    uint   in_use;        // 1=已分配, 0=空闲
    struct block_hdr *next; // 空闲链表中的下一个空闲块
    struct block_hdr *prev; // 空闲链表中的上一个空闲块（双向链表）
} block_hdr_t;
```

**关键约束**:
- `size` 必须始终 ≥ `sizeof(block_hdr_t)`
- 所有分配大小向上对齐到 8 字节
- `malloc(n)` 返回的指针是 `(char*)hdr + sizeof(block_hdr_t)`

### 2. 实现堆扩展 (lib/xv6_stdlib.c)

```c
static block_hdr_t *heap_extend(uint min_size)
```

- 计算需要申请的页数（`PGSIZE = 4096`），调用 `sbrk` 获取新内存
- 将新内存构造为一个大的空闲块，插入空闲链表尾部
- 尝试与前一个空闲块合并（如果紧邻）
- 返回新空闲块指针，失败返回 0

### 3. 实现 First-Fit 分配 (lib/xv6_stdlib.c)

替换 `x6_malloc` 的实现：

```c
void* x6_malloc(uint n)
```

算法流程：
1. 将 `n` 上舍入到 8 字节对齐，加上 header 大小得到 `total`
2. 遍历空闲链表，找到第一个 `size >= total` 的空闲块
3. 如果找到的块足够大（剩余部分 > `MIN_SPLIT_SIZE`），将其**分裂**为两块：分配块 + 新空闲块
4. 将分配块从空闲链表摘除，`in_use = 1`
5. 空闲链表遍历完未找到 → 调用 `heap_extend`，重试

**提示方向**:
- `MIN_SPLIT_SIZE = sizeof(block_hdr_t) + 8`，太小的剩余块不必分裂（避免碎片增殖）
- 分裂时新空闲块从当前块尾部切割：`new = (block_hdr_t*)((char*)hdr + total)`

### 4. 实现 Free 与合并 (lib/xv6_stdlib.c)

替换 `x6_free` 的实现：

```c
void x6_free(void *p)
```

算法流程：
1. 从 `p` 反推出 `hdr = (block_hdr_t*)p - 1`
2. 断言 `hdr->in_use == 1`（双重释放检测）
3. `hdr->in_use = 0`，将块插入空闲链表头部
4. **向后合并**: 检查 `(char*)hdr + hdr->size` 处的块是否空闲，若是则合并
5. **向前合并**: 遍历空闲链表找前驱块，若与本块紧邻则合并（或维护额外反向指针加速）

**合并示意**:
```
释放前: [用中 A][空闲 B][用中 C][空闲 D]
释放 A: [空闲 A][空闲 B][用中 C][空闲 D]
合并后: [    空闲 A+B   ][用中 C][空闲 D]
```

### 5. 编写测试程序 (user/malloctest.c)

验证以下场景：

| 测试 | 验证内容 |
|------|---------|
| 基础分配/释放 | malloc(n) 返回非空，free 后可重新分配 |
| 内存对齐 | 返回地址 % 8 == 0 |
| 合并效果 | free A, free B（A 和 B 相邻），再 malloc(A+B 大小) 成功且不扩堆 |
| 大量小对象 | 循环 malloc/free 1000 次，堆大小不持续增长 |
| 零字节分配 | `malloc(0)` 返回非 NULL 的合法地址（glibc 行为） |

## 涉及的 OS 概念

| 概念 | 在本实验中的体现 |
|------|----------------|
| `sbrk` 系统调用 | 分配器向内核申请内存的唯一接口 |
| 堆 (Heap) | 进程数据段向上增长的动态内存区域 |
| 内存对齐 | x86 要求 4/8 字节对齐，否则可能触发总线错误 |
| 碎片化 | First-Fit 会产生外部碎片；合并策略是关键对抗手段 |
| 链表操作 | 空闲块用双向链表管理，插入/删除均为 O(1) |
| 隐式元数据 | header 紧贴在 payload 之前，用户感知不到其存在 |

### 验证目标

| 目标 | 预期行为 | 观察方式 |
|------|---------|---------|
| 基础分配 | `malloc(100)` 返回非空，数据可写入 | malloctest 输出 PASS |
| 地址对齐 | 所有返回地址 % 8 == 0 | malloctest 逐个检查并输出 |
| 内存复用 | free 后再 malloc 不无限扩堆 | 调用 `sbrk(0)` 检查 break 是否稳定 |
| 边界合并 | 释放相邻块后能分配更大块 | 合并测试输出 `merge OK` |
| 无悬挂指针 | 已释放内存不再被程序使用 | 代码审查 |

### 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| `malloc` 返回 NULL | `heap_extend` 中 `sbrk` 失败 | 检查 xv6 堆上限（MAXVA/USERTOP） |
| 数据被 header 覆盖 | `malloc` 返回了 header 起始而非 payload 起始 | 返回 `(char*)hdr + sizeof(block_hdr_t)` |
| 无限扩堆 | 合并没生效，释放的块没被复用 | 检查 `in_use = 0` 和链表插入逻辑 |
| 对齐错误 | 分裂后新块的地址没有对齐 | 确保 `total` 经过对齐计算 |
| 双重释放崩溃 | `free` 同一指针两次 | 在 `free` 开头断言 `hdr->in_use == 1` |

## 关键代码路径

- 分配入口: `lib/xv6_stdlib.c:x6_malloc()` → 遍历空闲链表 → 找到 → 分裂 → 摘除
- 扩堆路径: `x6_malloc()` → `heap_extend()` → `sbrk()` → 构造空闲块
- 释放入口: `lib/xv6_stdlib.c:x6_free()` → 标记空闲 → 插入链表 → 尝试合并
- 向后合并: `x6_free()` → 检查下一块 `in_use == 0` → 修改 `size`，摘除相邻块

## 设计权衡

| 方面 | 原始实现（sbrk 不回收） | First-Fit 空闲链表 |
|------|----------------------|--------------------|
| 分配速度 | O(1) 但内存持续增长 | O(n) 遍历链表 |
| 内存利用率 | 无法复用，持续浪费 | 释放后可复用 |
| 实现复杂度 | 极简 | 中等（需正确维护链表） |
| 碎片控制 | 无碎片（仅增长） | 外部碎片存在，靠合并缓解 |
| 真实系统对比 | - | glibc 使用分区 bin + 合并，更复杂 |

## 进阶挑战

- [ ] 实现 **Best-Fit** 策略：遍历全表找最小合适块，比较与 First-Fit 的碎片化程度
- [ ] 实现 **Buddy System**（伙伴系统）：按 2 的幂次分配，合并时只与"伙伴"合并
- [ ] 添加 `realloc(p, new_size)`：尝试原地扩展，失败则重新分配并复制
- [ ] 添加 **内存泄漏检测**：在分配时记录调用栈，程序退出时报告未释放块
- [ ] 添加 **线程安全**（当实现多线程后）：为空闲链表加自旋锁
