# Lab: Interactive Shell Line Editor (Tab + History)

[English](../../README.md)

难度: ★★★☆☆

## 设计初衷

xv6 默认 shell 使用非常简化的 `gets` 按行读取：可以输入和退格，但没有命令历史，也没有自动补全。

本实验把 shell 升级为一个小型交互式行编辑器，包含两个实用功能：

- `Tab` 自动补全命令/文件名前缀
- 上下方向键切换历史命令

核心问题：*"在遵循 xv6 控制台行规约的前提下，如何实现更友好的 shell 交互？"*

## 前置知识

- 理解 `user/sh.c` 的主循环：打印提示符 -> 读入一行 -> 解析 -> 执行
- 知道方向键的转义序列（`ESC [ A`、`ESC [ B`）
- 理解 `kernel/kbd.c` 中方向键对应的 xv6 键码（`KEY_UP`、`KEY_DN`）
- 理解 `kernel/console.c` 的输入缓冲语义（`input.r/w/e`）
- 熟悉 xv6 用户态文件 API（`open`、`read`、`close`、`struct dirent`）
- 熟悉固定长度缓冲区的安全编辑（光标索引、字符串结尾、边界检查）

## 实验内容

### 1. 定位输入路径并替换 `gets` (user/sh.c)

在 `main` 中，shell 当前通过 `gets(buf, nbuf)` 读入命令行。

请实现自定义按字节读取函数（如 `readline`），从 fd 0 逐字节读取并直接处理控制按键。

约束：
- 保持 xv6 提示符风格（`$ `）
- EOF 或读取失败返回 `-1`，保持现有退出语义

### 2. 修正控制台唤醒粒度 (kernel/console.c)

这是本实验最容易被忽略的关键点。

xv6 默认控制台输入主要在换行/EOF 时唤醒读者；在这种模式下，用户态 `readline` 无法实时收到方向键和 Tab 事件。

需要调整 `consoleintr`，使输入可以按字符唤醒（character-wise wakeup），同时保持控制台稳定。

### 3. 添加固定大小历史环形缓冲区 (user/sh.c)

添加内存中的历史记录结构，例如：

```c
#define HIST_MAX 16
#define HIST_LINE_MAX 100
```

需要维护：
- 历史总条数
- 按上下键浏览历史时的当前索引

规则：
- 空命令不入历史
- 连续重复命令可选去重
- 仅保留最新 `HIST_MAX` 条

### 4. 解析方向键输入 (user/sh.c)

需要同时支持两类输入：
- xv6 键码：`KEY_UP`、`KEY_DN`
- 可选 ANSI 转义序列：`ESC [ A`、`ESC [ B`

原因：
- 在 xv6 的 QEMU 控制台中，方向键常常以 `KEY_UP/KEY_DN` 单值到达，而不是标准 ANSI 三字节序列。

ANSI 三字节序列：
- `ESC [ A`：上键（更旧命令）
- `ESC [ B`：下键（更新命令）

切换历史时：
- 用选中命令替换整行编辑缓冲
- 重绘当前行，保证屏幕与内部缓冲一致

提示：
- 简单重绘即可：输出 `\r`、提示符、清尾空格，再打印当前缓冲

### 5. 实现 Tab 前缀补全 (user/sh.c)

对当前 token 前缀（通常是行尾 token）做匹配：
- shell 内建命令（`cd`、`exit`）
- 当前目录（`.`）中的可执行/文件名

建议行为：
- 唯一匹配：直接补全并追加空格
- 多个匹配：换行打印候选项，再重绘提示符与当前输入
- 无匹配：不做修改

约束：
- 严禁越界写入输入缓冲区
- 仅用户态实现（不改内核）

### 6. 保持原有编辑能力 (user/sh.c)

修改后仍需正确支持：
- 普通可打印字符输入
- 退格（`\b` / DEL）
- 回车提交（`\n`）

可选：
- `Ctrl-U` 清空整行

### 7. 用交互场景验证功能

在 xv6 中按清单手工验证：
- 重复执行命令后，上下键切换历史
- 输入前缀后按 Tab，覆盖唯一匹配与多候选场景
- 混合测试：先输入再切历史再执行

### 验证目标

| 目标 | 预期行为 | 观察方式 |
|------|---------|---------|
| 历史切换正确 | 上键显示更旧命令，下键回到更新/当前空行 | shell 交互验证 |
| 唯一补全 | 前缀补全为完整命令/文件并追加空格 | 输入前缀按 Tab |
| 多候选补全 | 输出候选列表并恢复当前输入行 | 输入短前缀按 Tab |
| 缓冲区安全 | 长输入或频繁按键下不崩溃不乱码 | 压力交互测试 |
| 兼容性 | 原有命令解析执行不受影响 | 运行 `ls`、`echo`、管道命令 |

### 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 按上下键出现 `^[` 字符 | 转义序列解析不完整 | 先识别 `ESC`，再读 `[` 与最终码 |
| 按上下键/Tab 必须回车后才有反应 | 控制台仍是按行唤醒（换行才唤醒） | 修改 `kernel/console.c` 为按字符唤醒 |
| 在 QEMU 中上下键不生效 | 只处理了 `ESC [ A/B`，未处理 `KEY_UP/KEY_DN` | 在 `readline` 同时支持两类键输入 |
| 历史覆盖显示错乱 | 重绘时未清除旧行尾部 | 重绘前后补空格清尾再回写 |
| Tab 补全破坏输入缓冲 | token 边界或长度判断错误 | 精确计算 token 起点并做长度保护 |
| 组合按键后卡住 | ESC 分支下阻塞读取处理不当 | 仅在收到 `ESC` 后读取后续字节 |

## 关键代码路径

- `user/sh.c:main()` -> 提示符 -> `readline()` -> `parsecmd()` -> `runcmd()`
- `kernel/console.c:consoleintr()` -> 输入提交+唤醒 -> `consoleread()`
- `readline()` -> 字节分发：普通字符/退格/回车/Tab/ESC
- `readline()` -> 历史切换 -> 重绘
- `readline()` -> Tab 补全 -> 更新缓冲 -> 重绘

## 设计权衡

| 方面 | 原始 shell | 本实验 |
|------|-----------|--------|
| 控制台唤醒 | 主要按行 | 按字符（交互友好） |
| 输入模型 | `gets` 按行读取 | 按字节事件驱动解析 |
| 历史能力 | 无 | 固定容量环形历史 |
| 自动补全 | 无 | 基于前缀匹配（内建命令 + 当前目录） |
| 实现复杂度 | 低 | 中等，但仍完全用户态 |
| 内核改动 | 无 | 调整 `console.c` 唤醒行为 |
